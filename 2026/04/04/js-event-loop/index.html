JS 是单线程的

JavaScript 是单线程语言，意味着它同一时刻只能做一件事。所有代码都在一个线程里排队执行，不存在两段代码真正并行运行的情况。

这有历史原因。JS 最初是为浏览器设计的，用来操作 DOM。如果多个线程同时修改同一个 DOM 节点，会产生无法预测的冲突。单线程彻底避免了这个问题。

但单线程带来了一个显而易见的问题：如果某个操作很慢（比如从服务器请求数据），整个线程就会卡在那里，用户界面完全冻结，没法点击任何东西。

JS 的解决方案是异步 + 事件循环。

调用栈

调用栈(Call Stack)是 JS 引擎追踪”当前在执行哪个函数”的数据结构。每调用一个函数，就把它压入栈顶；函数执行完，就从栈顶弹出。这个概念比较好理解，这是大多数设计都会使用的数据结构。

调用栈清空，代表当前这一轮同步代码执行完毕。

任务队列

网络请求、定时器、用户点击这些操作，结果出来的时机是不确定的。JS 不可能在等它们的时候什么都不做，所以引入了任务队列（Task Queue）。

机制是这样的：

  1. 异步操作（如 setTimeout）被注册后，JS 引擎不等待，继续往下执行
  2. 当异步操作完成（定时器结束、请求返回、用户点击），对应的回调函数被放进任务队列
  3. 当调用栈完全清空后，事件循环从任务队列里取出第一个任务，压入调用栈执行
  4. 执行完再次清空，再取下一个，如此循环

这个”不断检查调用栈是否为空，为空就从队列取任务执行”的机制，就叫事件循环（Event Loop）。

回到 Bug：setTimeout(0) 做了什么

现在可以解释那个 bug 了。

没有 setTimeout(0) 时

lightGallery 触发 onAfterOpen 事件时，它自己还没执行完——写入 uuid.jpg 到 sub-html 的代码还在调用栈里排着。

我们的 handler 在这时执行，如果缓存命中，fetchMeta这个获取caption信息的函数几乎立即返回，然后caption被写进 sub-html。接着 lightGallery 继续执行，把 uuid.jpg 写进去，覆盖了我们的 caption。

第一次打开时没有这个问题，是因为 fetchMeta 要发网络请求，耗时几百毫秒。等请求回来时，LG 早已经把 uuid.jpg 写完了，我们再覆盖，顺序是对的。

这是一个典型的竞态条件（Race Condition）：同一段代码，因为快慢不同，执行顺序不同，结果也不同。

加了 setTimeout(0) 后

setTimeout(fn, 0) 不是”立刻执行 fn”，而是”把 fn 放进任务队列，等当前调用栈清空后再执行”。

0 毫秒的意义不是”等待时长为零”，而是”尽快，但必须在当前同步代码执行完之后”。这一个字之差，决定了在队列里的位置。

宏任务与微任务

实际上任务队列不只一个。JS 中的异步任务分为两类：

宏任务（Macrotask），也叫任务（Task）：

  • setTimeout / setInterval
  • I/O（网络请求完成、文件读取）
  • 用户事件（点击、键盘输入）
  • postMessage

微任务（Microtask）：

  • Promise.then / Promise.catch / Promise.finally
  • queueMicrotask
  • MutationObserver

执行顺序的规则是：

每次调用栈清空后，先把所有微任务队列清空，再取一个宏任务执行。

async/await 是什么

async/await 是 Promise 的语法糖，本质上还是微任务。

输出：A → C → B

await 之后的代码被包装成一个微任务，等当前同步代码（打印 C）执行完后，才从微任务队列取出执行（打印 B）。

为什么要这样？就你JS事多是吧

单线程 + 事件循环是一个权衡。

放弃多线程，避免了锁、竞态、死锁这一整套并发问题，代价是所有耗时操作必须异步化。事件循环让 JS 在等待异步结果的同时，仍然能响应用户操作，界面不会冻结。

宏任务和微任务的优先级划分也有实际用途：微任务（Promise）用于需要在当前操作”结算”之前完成的事，比如状态更新、数据转换；宏任务（setTimeout）用于真正的延后，让出线程给渲染引擎刷新界面。

浏览器的渲染（repaint）发生在每个宏任务之间，微任务不会触发渲染。所以如果你在微任务里不停地生产新微任务，渲染会被一直阻塞——这是一个实际的性能陷阱。
