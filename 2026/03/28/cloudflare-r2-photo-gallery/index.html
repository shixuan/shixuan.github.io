Cloudflare R2搭建相册方案

记录一下如何用 Cloudflare R2 + Workers 搭建一个个人相册，支持缩略图、瀑布流排版、灯箱查看，以及带鉴权的管理后台。

整体架构

架构分三层：

  1. 静态前端：Hexo 生成的两个页面——公开的相册浏览页、受保护的管理后台页
  2. Cloudflare Worker：处理所有动态 API 请求（列表、图片代理、上传、删除），绑定 R2
  3. Cloudflare R2 Bucket：存储原图和缩略图

Worker 只处理特定 API 路由，其余请求（包括静态页面本身）正常回源到 GitHub Pages。

1. R2 Bucket

在 Cloudflare Dashboard 创建一个 R2 Bucket。不需要开启 Public Access，图片通过 Worker 代理，不直接暴露 R2 URL。

原图和缩略图共享同一个 key，缩略图加 thumb/ 前缀存放。

2. Cloudflare Worker

一个 Worker 处理所有 API 路由，绑定 R2 Bucket（binding 名 PHOTOS）。

路由结构

GET /gallery/list — 图片列表（分页）

从 R2 列出所有对象，过滤出原图（排除 thumb/ 前缀），按上传时间倒序，返回分页结果：

R2 的 list() 不支持随机跳页，只能全量拉取后在 Worker 内切片。对于几百张图的个人相册完全够用。图片量上千时可以考虑用 KV/D1 维护一份索引。

GET /gallery/img/:key 和 GET /gallery/thumb/:key — 图片代理

接收 multipart/form-data，字段为 original（原图）和 thumb（缩略图），用 UUID 作 key 存入 R2：

3. 前端

相册页

加载策略是 Load More 累加 + 分区上限：每次加载一批（24 张）追加到画廊，超过上限（120 张）后出现分区导航。

和传统翻页相比，Load More 模式的每批图片会补全上一批未填满的末行，瀑布流视觉效果更好；分区上限保证 DOM 不会无限增长。

追加图片时用 Justified Gallery 的 norewind 方法增量排版，不重新渲染已有图片。

后台页

路径 /gallery/admin/，是一个普通的静态 Hexo 页面，包含两个 Tab：

Upload Tab：选择图片文件（支持多选），在浏览器端用 Canvas 生成缩略图，然后把原图和缩略图一起 POST 给 Worker。缩略图生成在浏览器端完成，不消耗 Worker CPU：

Manage Tab：分页加载已有图片，支持：

  • 点击图片查看大图（内置灯箱）
  • 勾选 Checkbox 批量删除
  • 单张图片右上角 ✕ 快速删除

5. 鉴权：Cloudflare Zero Trust

管理后台需要鉴权，使用 Cloudflare Zero Trust（Access）在边缘层拦截。

在 Zero Trust Dashboard 创建一个 Self-hosted Application：

  • Domain：example.com/gallery/admin（保护整个 admin 路径）
  • Policy：添加规则，例如 “Emails ending in @xxx.com” 或直接指定账户邮箱

配置后，访问 /gallery/admin/ 会先跳转到 Cloudflare Access 的登录页（支持 GitHub、Google 等 IdP），验证通过后才能访问管理页面。Worker 同样在这个保护范围内——即使用户直接请求 /gallery/admin/upload，Access 也会在请求到达 Worker 之前验证身份。

整个过程对前端透明，管理页面的 JS 不需要处理 token 或登录逻辑。

多此一举：Worker 内 JWT 校验

Zero Trust 已经在边缘层保护了 admin 路由，通常这就够了。如果想要 defense in depth（即使有人绕过 Access 直接打 Worker，Worker 自身也能拒绝），可以在 Worker 里额外校验 Cloudflare Access 注入的 JWT。

Access 会在每个已验证的请求里加一个 Cf-Access-Jwt-Assertion Header，内容是 RS256 签名的 JWT。Worker 可以：

  1. 从 https://<your-team>.cloudflareaccess.com/cdn-cgi/access/certs 获取公钥（JSON Web Key Set）
  2. 解析 JWT Header 找到对应的 kid
  3. 用 crypto.subtle.verify 校验签名
  4. 检查 aud（Application Audience）和 exp

在 upload 和 delete 的处理函数开头调用：

Worker 需要两个环境变量：CF_TEAM_DOMAIN（Zero Trust 团队域，格式 yourteam.cloudflareaccess.com）和 CF_AUD（Application Audience Tag，在 Access Application 详情页里）。

这一步是可选的——如果信任 Cloudflare Access 边缘层的防护，Worker 里可以省略 JWT 校验。
