Cloudflare R2 个人相册方案

用 Cloudflare R2 + Workers 搭建个人相册的完整记录，包含存储、API、前端两个页面及鉴权方案。

整体架构

三层：静态前端（Hexo 生成）→ Cloudflare Worker（动态 API）→ R2 Bucket（存储）。

Worker 只拦截特定路由，其余请求回源 GitHub Pages。

R2 Bucket

不开启 Public Access，图片通过 Worker 代理，不暴露 R2 URL。

原图、缩略图、caption 共享同一个 UUID key，前缀区分类型。

Cloudflare Worker

一个 Worker 处理所有路由，绑定 R2（binding 名 PHOTOS）和 Workers AI（binding 名 LLM）。

GET /gallery/list — 图片列表

从 R2 全量拉取对象列表，过滤原图（排除 thumb/、meta/ 前缀），按上传时间倒序，分页切片返回：

R2 list() 单次最多返回 1000 个对象（truncated: true 时需用 cursor 继续翻页），不支持随机跳页，只能全量拉取后在 Worker 内切片。几百张完全够用，上千张时考虑用 KV/D1 维护有序索引以避免每次全扫。

前端：相册浏览页

路径 /gallery/，Hexo 静态页面。代码结构：

  • gallery/gallery.js — 逻辑（独立 JS 文件，index.md 仅引用）
  • gallery/gallery.css — 样式（通过 after_render:html filter 注入 <head>）

依赖：

  • justifiedGallery（CDN）— 瀑布流布局
  • lightGallery v2（CDN 动态加载）— 灯箱查看

Hexo marked 渲染器会过滤 <link> type-6 HTML block，CSS 不能直接写进 .md，必须用 hexo.extend.filter.register('after_render:html', ...) 注入。同理 JS 文件也从 .md 里分离到独立文件，保持代码结构整洁。

参数 值
每批数量 16 张
单 section 上限 128 张（8 批）

触发机制：用 IntersectionObserver 监听当前已渲染图片的倒数第二行第一个元素，进入视口时触发下一批加载。比固定像素阈值更精准，用户还有最后一行可看时加载已在后台进行。

行定位：justifiedGallery 对 <a> 用绝对定位，style.top 相同的为同一行，去重排序后取倒数第二个。

灯箱：lightGallery v2

Icarus 内置的是 lightGallery v1（jQuery 插件，与 inferno 强耦合，维护者无意升级）。相册页在运行时动态加载 v2 UMD bundle，两者命名空间不冲突（v1 是 $.fn.lightGallery，v2 是 window.lightGallery）：

CSS 通过 inject-css.js 的 after_render:html filter 注入 <head>（v2 的 transitions 已内置在主 CSS 里，不需要额外引入 lg-transitions.min.css）。

动态模式（dynamic: true）：LG v2 不扫描 DOM <a> 标签，改用数据数组驱动，需要手动给每个 anchor 绑 click handler，并维护全局 index：

灯箱内跨批次无限滑动：在 lgAfterSlide 里检测当前 index 是否到达倒数第六张，到达则后台触发 loadMore()；jg.complete 后把新图片追加进灯箱，用户完全无感知：

触发点选在倒数第二张：用户还有最后一张可看，fetch 已在后台进行。由于追加用 push 而非 updateSlides，fetch 完成时机不影响已渲染的 slide，不需要更大的缓冲。

追加方案：直接 push 而非 updateSlides

LG v2 提供 updateSlides(wholeArray, index) 用于动态更新灯箱内容，但它会替换整个 galleryItems 数组并重渲染当前 slide 及 preload 范围内的 slides——用户正在看的图片会闪一下，体验很差。

工业界的标准解法是 append-only API（PhotoSwipe v5 的做法：numItems filter 动态上报，已渲染 DOM 完全不动）。LG v2 没有这个接口，但可以绕过去：

LG 是懒渲染的，preload 范围外的 slide 根本没有 DOM，只是 galleryItems 数组里的一个普通对象。直接 push 进数组不会触发任何渲染，对已渲染的 DOM 零影响：

这依赖 LG 内部懒渲染的实现，是非公开 API 的用法，但比 timing hack（推迟 updateSlides 调用时机）更干净，从根本上消除闪烁。

JG 重排与灯箱的协调：灯箱打开时追加图片若触发 JG norewind，DOM 重排会导致帧率下降和闪黑屏。用 _pendingJgRefresh 标志推迟：灯箱打开期间跳过 JG 重排，灯箱关闭（lgAfterClose）后补做 norewind。

灯箱打开时 scroll observer 也应停止（灯箱遮住页面，observer 触发无意义），关闭后恢复。通过 lgInstance.lgOpened 属性判断当前状态。

UI 显隐：点击图片区域切换工具栏、箭头、caption 的显隐（lg-ui-hidden class + opacity transition）。统一监听 click 事件即可，移动端浏览器在 touch tap 后会合成 click，不需要单独处理 touchend。listener 绑在 document 上（LG overlay 挂在 body，冒泡可达），LG 初始化时绑一次，lgBeforeClose 里移除：

Caption：lgAfterSlide 时 fetch /gallery/meta/:key，key 存在 lgInstance.galleryItems[index].key 里。metaCache 以 Promise 为值，防止同一 key 并发重复请求。resolve 时对比 lgInstance.index === index，不匹配则只更新 galleryItems[index].subHtml 不写 DOM，防止快速滑动时 caption 错位。

加载中 indicator：loadMore 触发时若灯箱已打开，在 .lg-toolbar 追加 “Loading…” 文字。移除时机在 jg.complete 的回调里（push 完成之后），而非 setLoading(false)（fetch resolve 即触发，早于 JG 排版完成）。

性能：preload: 3（预加载前后 3 张全尺寸图），speed: 300ms（切换动画）。滑动体感的主要瓶颈是全尺寸原图的首次网络请求，首次请求走 Worker → R2 无浏览器缓存；命中 Cache-Control: max-age=31536000 后近乎瞬间。

前端：管理后台

路径 /gallery/admin/，Zero Trust 保护，两个 Tab。

管理页代码结构同样分离：admin/admin.js、admin/admin.css 独立文件，index.md 只含页面骨架。

Upload Tab：多选图片 → 浏览器端 Canvas 生成缩略图（800px 宽，JPEG 0.75 质量）→ POST 原图 + 缩略图到 Worker。

Manage Tab：网格展示已有图片，支持：

  • 点击缩略图查看大图（自定义 lightbox，img 动态创建而非静态 <img>，避免 Icarus 全局 LG v1 在初始化时抓取到空图然后序列错乱）
  • Checkbox 批量删除（Promise.all 并行，按实际结果移除 DOM）
  • 编辑按钮打开 caption 弹窗，支持手动输入或一键 Auto-translate

v2 迁移后的问题与修复

UI 显隐失效

现象：点击灯箱大图，工具栏不隐藏。

原因：v1 时事件处理绑在 jQuery 插件内部，v2 需要手动管理。曾尝试用 touchend + tap 检测处理移动端，但移动端浏览器在 touch tap 之后会合成一次 click，导致 touchend 和合成 click 各触发一次 toggle，两次抵消，UI 状态不变（视觉上”闪一下”）。

修复：去掉 touchend handler，统一用 click 事件。合成 click 的存在说明浏览器在 LG 环境下不阻断点击合成，直接依赖它即可：

页面初始化时 loading flag 死锁

现象：页面加载后一直显示 “Loading…”，图片不出现。

原因：window.load 时先调 setLoading(true)（为了早点显示 loading 文字），这把 loading 标志置为 true；随后 loadMore() 开头的 if (loading) return 直接返回，实际不加载任何内容。

修复：初始化时只直接操作 DOM 元素显示 loading 文字，不调 setLoading()，不动 loading 标志：

鉴权：Cloudflare Zero Trust

在 Zero Trust Dashboard 创建 Self-hosted Application，Domain 指向 example.com/gallery/admin，Policy 限定邮箱或 IdP 账户。

访问 /gallery/admin/ 跳转到 Cloudflare Access 登录页，验证通过后注入 Cf-Access-Jwt-Assertion Header。Worker 可选择校验此 JWT（RS256，从 <team>.cloudflareaccess.com/cdn-cgi/access/certs 取公钥验签）作为 defense in depth，不校验也行，Access 边缘层已拦截未授权请求。
