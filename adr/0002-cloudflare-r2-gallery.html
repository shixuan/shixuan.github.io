ADR 0002 — Cloudflare R2 个人相册

Context

想给博客加个相册，几个硬约束：不要用第三方图床（容易跑路、限速、加广告）、不要自己运维服务器、管理后台必须有访问控制、要支持 lazy load + lightbox + 中英 caption。GitHub Pages 是纯静态，做不了上传/删除；普通图床又走不通后台。需要一个”静态前端 + 轻量动态 API”的组合。

Decision

三层架构，Cloudflare 把动态那一层和存储一起承包：

R2 不暴露 URL，所有图片走 Worker 代理，加长期缓存头。前端代码继续在 Hexo 里写，跟博客其他页面共用同一套构建。

Notes

Worker 单域名拦截

Worker 部署在 example.com 的 route 上，按 URL.pathname 决定动作：

  • /gallery/... → 走 Worker 内的 R2 / API 逻辑
  • 其他一律回源 GitHub Pages

这样不需要拆子域名、不需要 CORS 配置，相册 API 和博客静态资源在同一个 origin 下。但代价是 Worker 一定不能写错 — 任何 /gallery/list 之类的路由 typo 都会变成”穿透到 Pages 然后 404”，调试时容易迷惑。

R2 list() 不支持随机翻页

R2 的 list() API 单次最多返回 1000 对象，超过要靠 cursor 顺序翻页，没有跳页。每次拉某一页只能：全量扫一遍 → 排序 → 切片。

几百张图全扫毫无压力。上千张时这个写法会成为瓶颈，正确做法是用 KV/D1 维护一个有序索引，避免每次都全扫 — 等真的撞上再说。

原地替换图片

用 customMetadata 锁定排序，避免 R2 内置的 last-modified 时间戳在覆盖写后刷新，把被替换的图挤到列表顶部。

修过的图想重新上传，但要保住文件名、description 和列表顺序 — 这意味着不能换 key、不能动 meta/<key>、覆盖写还得不影响排序。前两点拿同一个 key put 就够了，第三点是真正的坑。

R2 对象的 o.uploaded 是 last-modified 时间戳，任何一次 put 都会被刷新。列表只要按它排序，被替换的图就会跳到顶部。

修复办法是把”创建时间”和”最后修改时间”分两个字段存：upload 时额外写一份 customMetadata.uploaded，作为不可变的逻辑创建时间；list() 加 include: ["customMetadata"]，排序改用这个值，回退到 o.uploaded 兼容老对象：

/gallery/admin/replace 路由先 head() 读原对象的 customMetadata.uploaded，覆盖写时把同一个值塞回去；meta/<key> 完全不动。文件名、上传时间、description、显示顺序四样全保住，只换图片内容。

这个分裂其实是把对象存储里”系统时间戳”和”业务时间戳”区分开 — 数据库里给每行加 created_at / updated_at 是同一回事，只是 R2 这层默认只提供 updated_at，created_at 得自己用 customMetadata 维护。

替换后浏览器看到旧图，用 ?v= cache-bust

Worker 给图片设了 Cache-Control: public, max-age=31536000, immutable — 这是基于”UUID 命名，内容永远不变”的契约。替换功能撕毁了这个契约（同 URL，内容换了），但浏览器还信着，本地 disk cache 里那条记录会被读上一年，Ctrl+F5 都没用（动态 JS 创建的 <img> 脱离了 hard refresh 作用域，照样吃 immutable）。

解决方案是把”新内容 = 新 URL”做出来。前面那两套时间戳里，o.uploaded（系统时间，每次 put 自动刷新）天然就是版本号 — 不用新增字段、不用维护计数器、本来就在 list() 返回里。

handleList 返回时把 o.uploaded 单独暴露成 version：

前端（公开 gallery + 管理后台）拼图片 URL 时统一带上 ?v=${version}：

Worker 的 handleImage 只 parse pathname，query string 完全不看 —— ?v= 是纯客户端层面的 cache key 操纵，对服务器逻辑零影响。

替换后浏览器看到的是个全新 URL（cache 字典里查不到），必须发请求；旧 URL xxx.jpg（没带 ?v=）的本地 cache 条目继续诚实地缓存着旧图，只是再也没人请求它，过些天被 disk cache 淘汰策略清掉。

这跟 webpack/vite 出来的 app.a3f2bc91.css 是同一个思路 —— 内容变了就换文件名（hash），让”新内容 = 新 URL”自动成立。query string 版本号是它的轻量变体：path 不变（Worker 不用改），靠 query 撑出 URL 差异。代价是 URL 长一点，收益是 immutable + max-age=1y 长缓存完整保留，CDN 命中率不受影响。

lightGallery v2

运行时动态加载 v2 UMD bundle，跟主题内置的 v1 共存 — v1 是 $.fn.lightGallery，v2 是 window.lightGallery，命名空间不冲突。

Icarus 内置的 lightGallery 是 v1，jQuery 插件，跟 inferno 强耦合，作者明确不打算升级。但相册页要的功能（懒加载、动态数据源、handler hook）只有 v2 才有。CSS 通过 inject-css.js 的 after_render:html filter 注入 <head>。

灯箱跨批次无限滑动

LG v2 的官方 API updateSlides(wholeArray, index) 会替换整个 galleryItems 并重渲染当前 slide 及 preload 范围内的 slides — 用户正在看的图片会闪一下，体验很差。

但 LG 是懒渲染的，preload 范围外的 slide 根本没 DOM，只是数组里一个普通对象。直接 push 进 galleryItems 不会触发任何渲染，对已渲染的 DOM 零影响：

依赖 LG 内部 lazy render 的实现细节，是非公开 API 的用法 — 但比 timing hack（推迟 updateSlides 调用时机）干净，从根本上消除闪烁。

同样思路在工业界的标准实现是 PhotoSwipe v5 的 numItems filter — append-only API。LG v2 没暴露这个接口，只能从内部状态绕。

Caption 自动翻译走 Workers AI

中文已填、英文为空时调 @cf/qwen/qwen3-30b-a3b-fp8 翻译；用户手动填的英文优先，模型不会覆盖。Prompt 里加了 /no_think 关掉 reasoning 模式，避免给一个短 caption 烧一堆推理 token；max_tokens: 30 强制截断。

缩略图

能在浏览器做就别推给 Worker — Worker 有 CPU 时间上限，图像处理是最容易撞上的那类活。

缩略图（800px 宽 JPEG 0.75 质量）在浏览器端用 Canvas 生成，跟原图一起 POST 上来，Worker 只管落 R2。
