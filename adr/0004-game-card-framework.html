ADR 0004 — Game Card 侧栏 mini game 框架

⚠️ Superseded by ADR 0005 — 本方案已废除。框架已从博客抽出独立为 npm 包 arcade-shelf，博客现在以 CDN + mount-point 方式集成。以下内容仅作为历史快照保留，代码已从 repo 删除。

Context

想给博客加点 playful 的东西 — 灵感来自 Crumpler 主页那个能直接玩的 Pong。具体要求：侧栏一个卡片列出可玩的 mini game，点击弹模态窗，canvas 跑游戏，关闭即销毁。

最重要的一条是：加新游戏不能改一堆文件，否则永远只会有一个 Pong。

第一版能跑，但写完 cc review 一遍发现一堆”看起来对、长期会出问题”的设计 — 加游戏要手动改 inject-css.js、Pong 在 144Hz 屏上球速翻倍、innerHTML 拼模态窗有 XSS 隐患、游戏脚本必须严格在 framework 之后加载……既然要做就一次做对，索性整理成框架。

Decision

SSR shell + 客户端 framework + 自描述游戏，三层职责完全解耦：

层 文件 职责
SSR shell widgets/game-card.jsx 渲染卡片 chrome，把 whitelist/order 序列化成 data-* JSON
Framework source/js/game-card.js GameRegistry 队列 stub、模态窗、RAF 合并渲染
游戏 source/js/games/<name>.js 自描述 { name, icon?, canvasSize?, init(canvas) }
自动注入 scripts/inject-css.js 扫 source/js/games/*.js 自动 inject <script defer>

加新游戏的完整步骤：drop 一个文件到 source/js/games/。不改 widget、不改 framework、不改 inject-css。

即使游戏脚本先于 framework 加载（defer 顺序变了、第三方插件重排了 body_end），也不会丢注册 — 第一个执行的游戏文件会创建 stub，后续游戏 push 进 _q。framework 加载时把 _q flush 进真正的 registry。

跟 GA 的 window.dataLayer = window.dataLayer || [] 是一个意思 — load order 解耦的标准 pattern。代价是 window.GameRegistry 永久占用了一个 global，但个人站点没人在意。

Per-card data-* 配置

_data/shared-widgets.yml 里写的 widget 配置（哪些游戏要展示、按什么顺序），需要在 client 端生效。SSR 那边把数组 JSON.stringify 进 data-* 属性：

Client 端 el.dataset.whitelist 读出来 JSON.parse 回数组，按 card 分别过滤排序。这样多个 sidebar 各自显示不同的游戏子集都行，配置完全在 yaml 里改，不碰代码。

比把配置塞进全局 window.__gameCardConfig 干净 — 配置跟 DOM 节点一对一绑死，多卡片场景天然解决。

Modal 用 createElement

第一版的模态窗是 modal.innerHTML = '<div>...<button>×</button>...' 拼出来的。游戏名直接拼进 HTML，理论上有 XSS 风险（即使现在游戏都是自己写的，也不留这种坑）。

改成 createElement 之后是几乎零成本顺手做几件事：游戏名走 textContent 自动消除 XSS、按钮加 aria-label、modal 加 role="dialog" + aria-modal="true"、打开时 close button 自动 focus、关闭时 focus 还给打开前的元素。

innerHTML 写起来快 5 行代码，但每加一个 a11y 属性就要回去改字符串，越来越烂。createElement 一开始多两屏，但每个 attribute 都是独立可改的。

games/ 目录自动扫描

注意是 sort() — 给一个稳定的注入顺序。queue-stub 让顺序不影响正确性，但稳定顺序对调试和 cache 友好。

hexo server watch 模式下加新文件不会自动 pick up（这套扫描在脚本 init 时跑一次），要重启 server。生产 hexo generate 每次都重扫，无影响。
