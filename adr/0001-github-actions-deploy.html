ADR 0001 — GitHub Actions 自动部署

Context

博客有两个仓库：

  • 源仓（私有）— Hexo 项目本体，包含 source/_posts/、主题配置、deploy 脚本等
  • Pages 仓（公开）— shixuan/shixuan.github.io，存放 hexo generate 产出的静态文件，GitHub Pages 直接服务

源仓大体应该要私有(?) - 主要考虑的点是config文件能不能暴露出来：虽然扫了一遍里面没有什么值得隐藏的，但是考虑到之前commit历史中包含了OAuth app的secret(懒得清了)以及以后可能涉及到的扩展，最终还是决定私有。 但Pages 仓必须公开 — GitHub Pages 免费版只服务 public repo。

之前每次发文都得本地 hexo clean && hexo generate && hexo deploy，三个命令稳定但烦。想要的是：push 到源仓 main → 自动构建 → 自动推到 Pages 仓，写完文章就能去刷视频了。

Decision

GitHub Actions 跑构建 + 部署，用 SSH deploy key 把构建产物推到 Pages 仓：

触发只有一种 — push: branches: [main]。手动触发都不需要，写完文章 push 一下就行。

Notes

私有源 + 公开 Pages：两个 repo 之间走 SSH deploy key

跨 repo 推送要鉴权。两条路：

方案 优点 缺点
Personal Access Token (PAT) 配置简单 权限粒度粗，过期要重发
SSH deploy key per-repo，scope 最小 要生成密钥对、分别配置

走的是 SSH deploy key — 给 Pages 仓配一对密钥，公钥加到 Pages 仓的 Deploy Keys（Write access），私钥存到源仓的 Actions secrets 里 (HEXO_DEPLOY_PRI)：

ssh-keyscan github.com >> known_hosts 是关键，否则 git push 会卡在 “Are you sure you want to continue connecting”。

_config.yml 的 deploy 段配 SSH URL 而不是 HTTPS：

Theme 走 npm dep，不再 checkout submodule

deploy.yml 里那段注释掉的 “Checkout Icarus theme” 是历史遗迹 — 当年主题作为 git submodule 引入，CI 必须单独 checkout 才能拿到。现在 hexo-theme-icarus: 6.1.1 是 npm 依赖，npm install 自动拉取，CI 步骤少一截。

代价是没法直接改主题文件 — 但这个约束反而好，自定义需求都通过 ADR 0003 的 monkey-patch 走，主题升级路径完整保留。

图片放仓库根目录，build 时复制到 source/img/

仓库根有一个 img/ 目录，本地仓库里 source/img/ 不存在 — 是 CI 临时拼出来的。Hexo 把 source/ 整个当输入，source/img/ 会被原样复制到 public/img/ 服务出去。

把 img 留在根目录的好处：clone 仓库一眼能看到资源；坏处是新人不知道为什么要这个 copy 步骤。属于个人习惯，不强推。

FORCE_JAVASCRIPT_ACTIONS_TO_NODE24：第三方 action 的运行时兼容开关

GitHub Actions 平台正在从 Node 20 runtime 过渡到 Node 24，过渡期里部分 action 已经要求 Node 24，但 runner 默认还在 Node 20。设置这个 env var 强制所有 JS-based action 跑在 Node 24 上：

提前 opt-in 比等 GitHub 改默认值再被动应付舒服 — 出问题现在能从容修，不会哪天突然 build 全红。

hexo-deployer-git 的工作机制

npx hexo deploy 调用 hexo-deployer-git 插件，干这几件事：

  1. 拿 _config.yml 的 deploy.repo / branch
  2. 在 .deploy_git/ 里 clone（或复用）目标分支
  3. 把当前 public/ 内容覆盖进 .deploy_git/
  4. git add . && git commit -m "Site updated: ..." → SSH push

每次部署是单个 commit 覆盖式推送，Pages 仓的历史只有一行行 “Site updated”，Pages 仓本质是构建产物的镜像，不需要源码commit记录。
