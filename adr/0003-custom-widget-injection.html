ADR 0003 — 不 eject Icarus 主题加自定义 widget

Context

Icarus 是 npm 包装在 node_modules/ 里的。按官方文档加自定义 widget 只有一条路：eject 整个主题。eject 之后改东西自由了，但 npm 升级路径也彻底没了 — 之后每次主题更新都得手动 merge。

想要的是反过来：自定义 widget 文件放项目根目录，主题继续以 npm 依赖存在，配置接口和官方 widget 完全一致（shared_widgets.yml + _config.icarus.yml）。

Decision

Monkey-patch hexo-component-inferno 的 view.require：widget JSX 放在 widgets/<name>.jsx，启动脚本拦截 widget/<type> 这种 lookup，先查项目目录，找不到再 fallback 到原始 resolver。_data/shared-widgets.yml 加条目，_config.icarus.yml 把 key 列进 shared_widgets。

顺带把站点全局 CSS/JS 的注入方式也统一了 — 之前散在好几个地方写死，现在集中到 scripts/inject-css.js。

Notes

为什么要替换整个 view.require 而不是 wrap

最初想法是 wrap：保留原 view.require，只在前面插一段自定义查找逻辑。但读源码才发现内部的 _resolve 是 closure 捕获的，没法原地改 — 你换 view.require._resolve 没用，原函数引用的是闭包里的那份。

non-custom 路径必须委托回 originalRequire

第一版的实现里 fallback 直接走 require(originalResolve(filename))，绕过了 originalRequire 本身。看起来等价，但 originalRequire 内部可能做 wrapping、caching、error handling，绕过去就把这些都丢了。

委托回 originalRequire(filename) 是 monkey-patch 的基本礼貌：只接管你确实关心的 case，其余原样转发，不偷偷篡改副作用。

entry key vs widget type

两个名字差一个字符。shared_widgets: 列表里写的是 entry key（game_card），view.require 拿去 lookup 文件用的是 type 字段（game-card）。两套 namespace 名字相同会踩坑 — 更糟的是写错也能跑（直到某天加了第二个 widget 才暴露）。

约定下来：entry key 用 snake_case，type 跟文件名用 kebab-case。靠命名规则自我约束，没有更好的办法。

资源注入分两种场景

场景 用什么
全局每页都注入 hexo.extend.injector.register('head_end' | 'body_end', ...)
Per-page 资源（如 gallery / admin） hexo.extend.filter.register('after_render:html', ...) 替换 </head>
目录扫描自动注入（如 game card 的 games/） fs.readdirSync + injector loop

injector 不支持按路径过滤，所以 per-page 资源只能走 filter；filter 又不能用在每页都要的资源（重复替换有性能问题）。两套并存。
